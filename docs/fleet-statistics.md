# Anonymous fleet statistics

Fleet statistics answer two operational questions without turning FTW into a
cloud-dependent product:

1. How many distinct FTW installations have checked in recently?
2. Which public driver/core/optimizer versions are installed and healthy?

The feature is explicit opt-in and disabled by default. Local control, driver
execution, updates, and boot do not depend on a successful heartbeat.

## Payload boundary

Each installation gets a random 128-bit ID persisted in its local `state.db`.
It is not derived from a person, site name, hardware, network address, Nova
identity, or remote-access identity.

The schema contains only:

- anonymous installation ID and schema version;
- FTW core version and stable/beta/edge/native channel;
- optimizer version, transport, protocol version, and coarse status;
- public FTW driver catalog ID, version, source (`managed` or `bundled`),
  host-API range, capability kinds, instance count, and coarse status;
- whether the configured site-meter driver currently has healthy telemetry;
- a count of configured drivers that have no public catalog identity.

It explicitly excludes raw power/energy samples, forecasts, prices, household
configuration, site or person names, user-chosen driver names, serial numbers,
device IDs, MAC/IP addresses, hostnames, endpoints, credentials, logs, and error
messages. The builder joins telemetry health to local names internally and
drops those names before validation and serialization.

Custom local Lua-driver IDs are operator-authored and may themselves be
identifying. FTW therefore reports only their count, never their ID, filename,
version, capabilities, or health.

Use Settings → System → Fleet stats → Preview, or:

```bash
curl http://localhost:8080/api/fleet_statistics/preview
```

The preview is generated by the same code path as the submitted payload.

## Client behavior

Enable it in `config.yaml` and restart FTW:

```yaml
fleet_statistics:
  enabled: true
  endpoint: https://relay.ftw.sourceful.energy/fleet/heartbeat
  interval_h: 24
```

FTW submits immediately after start and then at the configured interval. HTTP,
DNS, validation, or collector failures are ignored by the control system. A
manual `POST /api/fleet_statistics/submit` is available when enabled.

## Collector behavior

Enable collection on `ftw-relay` with a persistent path and a strong admin
token:

```bash
FTW_FLEET_STATS_FILE=/var/lib/ftw-relay/fleet-statistics.json \
FTW_FLEET_ADMIN_TOKEN='<random-secret>' \
ftw-relay [other flags]
```

`POST /fleet/heartbeat` strictly validates the schema and is rate-limited per
source IP. The source IP is necessarily visible to the HTTP server and used in
memory for throttling, but it is never written to the fleet store. Records are
keyed only by anonymous installation ID; the newest heartbeat replaces the
previous payload. Entries expire after 30 days and the store is capped at
100,000 installations.

Read the aggregate with:

```bash
curl -H 'Authorization: Bearer <random-secret>' \
  https://relay.ftw.sourceful.energy/fleet/stats
```

The response contains active installation counts for 24 hours and 7 days, a
`healthy_24h` count (defined as a fresh/healthy configured site meter), core channels/versions, optimizer statuses/versions,
and per-driver active installations, configured instances, healthy instances,
and installation counts per version. It does not expose installation-level
records.

These are approximate operational statistics, not a billing or identity
system. Opt-out installations are invisible, multiple installs by one person
count separately, and a re-created state database receives a new anonymous ID.
